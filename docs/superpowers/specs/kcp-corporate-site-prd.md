## 1. ドキュメント概要

### 1.1 目的

このドキュメントは、株式会社KCPのコーポレートサイト（コーポレートページ + ブログ）の要件と設計意図を記録することを目的とする。後から読み返したときに「何を作ったか」と「なぜそう決めたか」が両方わかる内部リファレンスとして機能する。

### 1.2 スコープ

- **対象:** Phase 1（MVP）の実装要件
- **含む:** コーポレートランディングページ、ブログ機能、SEO基盤、Cloudflare Pages デプロイ
- **含まない:** Phase 2以降の機能（[Section 9](#9-将来計画phase-2以降) に記載）

## 2. ビジネスコンテキスト

### 2.1 KCPとは

株式会社KCPは、AIエージェントを活用した業務代行（BPO）を主軸とする企業。AI時代において企業が本業に集中できる環境を提供することをミッションとする。

**主要サービス:**
- SNS運用代行（AI自動化活用）
- Web・アプリ受託開発
- AI活用研修
- メディア運営（ブログ）

### 2.2 なぜ今このサイトを作るのか

3つの目的がある：

1. **法人実在性の証明** — 案件先・エージェント・紹介先が「信頼できる会社」と判断できる最低限の情報を提供する。名刺代わりとして機能する。

2. **SEO資産の集約** — 既存ブログ（Life Optimizer）のAI活用・業務効率化系コンテンツをKCPドメインに移植し、SEO評価をKCPのドメインに蓄積する。

3. **アフィリエイト収益の維持** — 移植コンテンツのアフィリエイトリンクを同一アソシエイトIDで継続利用する。

### 2.3 ターゲットユーザーと閲覧シナリオ

| ユーザー | 閲覧目的 | 見てほしいもの |
|---------|---------|--------------|
| 天野さんルートの案件先（DIY業界） | KCPが実在する会社かを確認 | Company・Service セクション |
| フリーランスエージェント（レバテック等） | 会社概要・事業内容の確認 | MVV・Company セクション |
| 紹介経由の潜在クライアント | 信頼性・専門性の判断 | Strength・Service セクション |
| ブログ経由のオーガニック流入 | AI活用・業務効率化の情報収集 | ブログ記事 |

## 3. 設計思想・原則

### 3.1 MVVとの整合性

サイト自体がKCPのValueを体現する形で設計する。

- **シンプル・小さく・速く** — ページ数を最小化（コーポレートは1ページ完結）。バックエンド不要。Cloudflare Pages の Static Site で高速配信。
- **複利で積み上げる** — ブログをSEO資産として継続的に積み上げる構造。記事はMarkdownで書き、Git管理することで将来的な移行コストをゼロに近づける。
- **常に進化する** — Phase 1は最低限のMVPに絞り、実績・フィードバックをもとにPhase 2以降を判断する。

### 3.2 コスト最小化の原則

初期コストをゼロに近づけ、変動コストも極小化する。

- ホスティング: Cloudflare Pages 無料プラン（無制限帯域幅・リクエスト）
- CMS: 不要。Markdownファイルを直接編集しGitでデプロイ。
- お問い合わせフォーム: Google Forms（無料）へ外部リンク。バックエンド実装なし。
- ドメイン代のみが実費（年間 ¥2,000〜6,000 程度）

### 3.3 技術選定の大方針

- **Static Site Generation（SSG）優先** — ランタイムサーバーを持たない。セキュリティリスクが低く、運用コストがゼロ。
- **依存関係を最小限に** — CMS、認証、データベース、バックエンドAPIを一切持たない。
- **コンテンツはMarkdownファイルで管理** — ローカルのObsidianで記事を書き、Gitにプッシュするだけでデプロイが完結する。

#### Section 1: Hero（ファーストビュー）

**表示内容:**
- 会社名: 株式会社KCP
- タグライン（= Mission）: 「AI時代に必要な業務を引き受け、お客様が本業に集中できる環境を提供する。」

**なぜこの設計か:**
タグラインにMissionをそのまま使うことで、会社の存在意義を最初の3秒で伝える。CTAは2つ（問い合わせ + 探索）にすることで、「今すぐ連絡したい人」と「もっと知りたい人」両方の導線を確保する。

#### Section 2: Service（事業内容）

**表示内容:**
4枚のカードUI。各カードにアイコン + タイトル + 1〜2行の説明文。

| サービス | 説明 |

**なぜこの設計か:**

**表示内容:**

- **Vision**: AIエージェントを駆使した高効率・低コストの代行業で、世界市場に通用するBPO企業を築く。

  1. シンプル・小さく・速く
  2. 複利で積み上げる

**なぜこの設計か:**
Mission と Vision はブロック（カード）で目立たせ、Values は3カラムに並べることで「3つある」という構造が視覚的に伝わる。エージェント・案件先が「この会社はどういう会社か」を深く知りたいときに参照するセクション。

### 4.4 ブログ機能

**URL構造:**

| ページ | URL |
|--------|-----|
| 記事一覧 | `/blog/` |
| 個別記事 | `/blog/[slug]/` |
| タグ一覧 | `/blog/tags/` |
| タグ別一覧 | `/blog/tags/[tag]/` |
| RSSフィード | `/rss.xml` |

**なぜサブディレクトリ方式か:**
サブドメイン（`blog.kcp.co.jp`）と比較して、サブディレクトリ（`kcp.co.jp/blog/`）の方がドメイン全体にSEO評価が蓄積されるため。

**記事管理:**
- コンテンツ: Markdownファイル（`src/content/blog/*.md`）
- フロントマター: タイトル・説明・公開日・タグ・著者・draft フラグ・featured フラグ
- 編集フロー: Obsidianで記事を書く → Gitにプッシュ → Cloudflare Pagesが自動ビルド・デプロイ

**ブログ機能詳細:**

| 機能 | 詳細 | なぜ |
|------|------|------|
| タグフィルタ | URLパラメータ方式（`/blog/tags/[tag]/`） | SSGと相性が良い。URLが共有・ブックマーク可能。JS不要。 |
| 目次（ToC） | 記事ページの右サイドバーに表示（デスクトップのみ） | Astroの `render()` が headings を自動生成するため追加ライブラリ不要 |
| OGP / メタタグ | 記事ごとにtitle・description・og:imageを設定可能 | SNSシェア時に記事が正しく表示されるために必要 |
| RSS | `/rss.xml` に全記事のフィードを生成 | RSSリーダーユーザーへのリーチと、将来的なコンテンツ配信基盤として |
| 読了時間 | 記事一覧・記事ページに表示（日本語換算: 500文字/分） | ユーザーが読む前に時間を見積もれるとエンゲージメントが上がる |
| Sitemap | `/sitemap-index.xml` に自動生成 | 検索エンジンのクロールを促進するために必要 |

## 5. 技術アーキテクチャ

### 5.1 技術スタック

| カテゴリ | 技術 | バージョン |
|---------|------|-----------|
| フレームワーク | Astro | 4.x |
| スタイリング | Tailwind CSS | 3.x |
| 言語 | TypeScript | strict モード |
| コンテンツ管理 | Astro Content Collections（Markdown） | — |
| テスト | Vitest | 1.x |
| 画像最適化 | sharp | 0.33.x |
| サイトマップ | @astrojs/sitemap | — |
| RSSフィード | @astrojs/rss | — |
| ホスティング | Cloudflare Pages | — |
| リポジトリ | GitHub（プライベート） | — |

> 各パッケージの実際のバージョンは `package.json` / `package-lock.json` を正とする。上記はメジャーバージョンの設計意図のみを記録する。

### 5.2 技術選定の理由

**Astro を選んだ理由:**
コンテンツ中心のサイトに対して、Astro は "Zero JavaScript by default" を原則とする。ReactやVueのようにコンポーネント単位でJavaScriptをバンドルしないため、ページ速度が速く、SEOに有利。また、Content Collectionsという組み込みのコンテンツ管理機能があり、外部CMSなしでMarkdownブログを構築できる。

**Tailwind CSS を選んだ理由:**
クラス名だけでデザインが完結するため、コンポーネントファイル内でスタイルとマークアップが分離されず、読みやすい。ダークテーマの実装が `dark:` prefix で簡潔に書ける。カスタムデザインシステム（カラーパレット等）をconfig一箇所で管理できる。

**Cloudflare Pages を選んだ理由:**
無料プランでも帯域幅・リクエスト数が無制限。GitHubプッシュでの自動デプロイが設定不要で使える。SSLも自動発行。他の選択肢（Vercel・Netlify）と比較しても、無料プランの制限が最も少ない。

**CMS（Headless CMS等）を使わない理由:**
記事はMarkdownファイルで管理し、GitでバージョンコントロールすることがKCPの運用スタイル（Obsidian → Git → 自動デプロイ）に最適。外部CMSを持つとベンダーロックインと月額コストが発生する。

### 5.3 出力方式

- `output: 'static'`（Static Site Generation）
- ランタイムサーバーなし。全ページをビルド時に静的HTMLとして生成。
- Cloudflare Pages のビルドコマンド: `npm run build`
- 出力ディレクトリ: `dist/`

### 5.4 ディレクトリ構成（概要）

```
kcp-site/
├── src/
│   ├── components/    # Astroコンポーネント
│   ├── content/       # Markdownブログ記事
│   ├── layouts/       # ページレイアウト
│   ├── pages/         # URLに対応するページ
│   ├── styles/        # グローバルCSS
│   └── utils/         # ユーティリティ関数
├── public/            # 静的ファイル（favicon、robots.txt等）
├── astro.config.mjs
├── tailwind.config.mjs
└── package.json
```

## 6. デザイン仕様

### 6.1 カラーパレット

| 用途 | カラー | Hex |
|------|--------|-----|
| ページ背景（最も暗い） | Base | `#0a0a0a` |
| カード背景 | Base-100 | `#141414` |
| セクション背景（青みがかり） | Base-200 | `#1a1a2e` |
| アクセント（メイン） | Accent | `#4a9eff` |
| アクセント（ホバー時） | Accent Light | `#7ab8ff` |
| アクセント（押下時） | Accent Dark | `#0066cc` |
| アクセント背景（薄） | Accent Muted | `#1a3a5c` |
| 本文テキスト | Text Primary | `#f0f0f0` |
| サブテキスト | Text Secondary | `#a0a0b0` |
| 非アクティブ・注釈 | Text Muted | `#606070` |
| ボーダー | Border | `#2a2a3e` |

**なぜダークテーマか:**
KCPのターゲット（テック系・AI活用層）に親和性が高い。テック系SaaS・スタートアップのサイトはダークテーマが多く、業界の文脈に合っている。また、信頼感・モダン感を与えやすい。

### 6.2 タイポグラフィ

| 用途 | フォント | 提供 |
|------|---------|------|
| 日本語テキスト | Noto Sans JP | Google Fonts |
| 英語・数字 | Inter | Google Fonts |
| コードブロック | JetBrains Mono（または Shiki デフォルト） | Google Fonts |

- 本文: 16px、行間 1.8
- 見出し: 太字（font-bold）、余白多め
- 日本語の `font-feature-settings: 'palt'`（プロポーショナルメトリクス）を有効化

### 6.3 レイアウト

| 設定 | 値 |
|------|-----|
| コンテンツ最大幅 | 1200px |
| ブログ記事最大幅 | 780px（読みやすさ重視） |
| セクションの縦余白 | 80px（上下） |
| ブレークポイント | 640px / 768px / 1024px |

**なぜ780pxか:**
読書に最適な1行の文字数は日本語で35〜50文字程度。Noto Sans JP 16pxで780px幅の場合、1行あたり約40文字になり、快適な読書体験が得られる。

### 6.4 アニメーション

| アニメーション | 実装 | タイミング |
|--------------|------|-----------|
| スクロールフェードイン | Intersection Observer | スクロールで要素が画面内に入ったとき |
| ホバーエフェクト | CSS transition | カードのリフト、ボタンの色変化 |

**アニメーションの原則:** 過度なアニメーションはUXを損ない、パフォーマンスに影響する。Intersection Observer を使ったスクロールアニメーションのみ採用し、自動再生・ループ系のアニメーションは使わない。

## 7. 意思決定ログ

### 7.1 MDXを使わない

- **選択:** Markdown + Content Collections（MDXなし）
- **理由:** Phase 1のブログ記事にはインタラクティブなコンポーネントを埋め込む必要がない。MDXは複雑性を増やし、Viteのビルドコンフィグに影響を与える。シンプルなMarkdownで十分。
- **将来の判断基準:** 記事内にインタラクティブなデモやグラフを埋め込みたくなった時点でMDXへの移行を検討する。

### 7.2 タグフィルタをURLパラメータ方式にした

- **選択:** `/blog/tags/[tag]/` の静的ページ生成方式
- **却下した選択肢:** クライアントサイドJSによるフィルタリング
- **理由:** SSGと相性が良い。URLが共有・ブックマーク可能で、タグページ自体がGoogleにインデックスされる（SEO上有利）。JavaScriptなしで動作するため、JS無効環境でも機能する。

### 7.3 お問い合わせをGoogleフォームにした

- **選択:** Googleフォームへの外部リンク
- **却下した選択肢:** Formspree（外部サービス）、メールアドレス直記載、バックエンド実装
- **理由:** Phase 1ではバックエンドを持たないという原則を徹底する。Googleフォームはスパムフィルタ・通知設定・回答管理がすべて無料で使える。メールアドレス直記載はスパムリスクがある。Formspreeは無料枠の月次送信数制限がある。

### 7.4 ToC生成にカスタムパーサーを使わない

- **選択:** Astroの `render()` が返す `headings` 配列を直接利用
- **理由:** Astroがビルド時にMarkdownを解析し、見出し一覧（`{ depth, slug, text }`）を自動生成してくれる。rehype-slugでIDを付与すれば、追加ライブラリなしでToC機能が完成する。

### 7.5 `applyBaseStyles: false` にした（Tailwind設定）

- **選択:** Tailwind統合の `applyBaseStyles: false`
- **理由:** AstroがTailwindのベーススタイルを自動注入するタイミングを制御できなくなるため、自前の `global.css` でスタイルの読み込み順序を明示的に管理する。

## 8. Phase 1スコープ外

以下の機能は意図的にPhase 1に含めない。

| 機能 | 除外理由 |
|------|---------|
| ロゴのプロ制作 | テキストロゴ「KCP」で代用可能。案件実績が増えてからブランディングに投資する |

| チーム紹介ページ | 現在は代表者1名のみ |
| 多言語対応（英語） | Phase 2以降。国内需要対応を優先する |
| CMS管理画面 | Markdownファイル直接編集で運用可能 |
| お問い合わせフォームのバックエンド | Googleフォームで十分 |
| アクセス解析（GA4） | Cloudflare Web Analyticsの無料機能で代替 |
| Playwright E2Eテスト | 静的サイトへの過剰投資。Vitest（utils）+ 手動確認で十分 |

- **ロゴ制作** — プロデザイナーに依頼
- **GA4連携** — Cloudflare Analyticsでは不足な詳細分析が必要になった時点
- **OGP画像の自動生成** — Satori + Resvgでブログ記事ごとのOGP画像を自動生成

### Phase 3: 多言語対応・グローバル展開

- **英語対応** — Astroの国際化機能（i18n）を使った英語版ページの追加
- **海外マーケティング** — 英語ブログコンテンツの追加

### Phase 4: ブログSEO強化

- **Life Optimizer からの記事移植** — AI活用・デジタルツール系記事を選定してKCPブログに移植
- **301リダイレクト設定** — 旧ドメインから新URLへのリダイレクト
- **構造化データ（JSON-LD）の拡充** — Article・Organization・BreadcrumbList の追加

### Phase 5: 機能拡張

- **ニュースレター** — Buttondown や ConvertKit との連携
- **コメント機能** — Giscus（GitHubディスカッションベース）
- **検索機能** — Pagefind（静的サイト向け全文検索）

### 10.2 記事更新フロー

```
Obsidian で記事を執筆（Markdown）
    ↓
src/content/blog/ に保存
    ↓
git push（GitHub）
    ↓
Cloudflare Pages が自動ビルド（約1〜2分）
    ↓
本番サイトに反映
```

フロントマターの `draft: true` を設定すれば、ビルドしても公開されない。記事のレビュー・修正期間中はこのフラグで管理する。

### 10.3 ドメイン設定（スコープ外）

ドメインの取得・設定は本要件定義書のスコープ外。Cloudflare Pages へのカスタムドメイン紐付けはCloudflareの管理画面から行う。
